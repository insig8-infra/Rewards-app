## Acceptance Criteria

- Contractor self-registration remains denied.
- OWNER can register contractors from Admin Web.
- ContractorID is generated by backend.
- Duplicate contractor mobile is blocked.
- Superseded after Manual UAT 1 / `DEC-043`: OWNER can update contractor photo/status only; contractor name and mobile are immutable after registration.
- OWNER can deactivate/unregister contractor.
- STAFF can list and view contractor details only.
- Server-side guards enforce OWNER-only mutations.
- Admin Web does not expose manual point-change controls.
- Typecheck, lint, tests, build, audit, and browser smoke checks pass.

## Implementation Notes

- Current ContractorID format is `CTR-000001`, generated from current contractor count. This is acceptable for the local foundation but should move to a database sequence or equivalent before production concurrency.
- Current contractor photo upload stores an image data URL in `photoUrl`; production should replace this with object storage/media upload once infrastructure decisions are made.
- Welcome SMS is recorded as `MOCK_PENDING_SMS_PROVIDER` in audit metadata. Real SMS and temporary MPIN behavior remain blocked by the OTP/SMS open questions.
- Deactivation updates both contractor and linked user status to `DEACTIVATED`; no hard delete is implemented.
- The live browser contractor list showed `Failed to fetch` because the API server was not running on `127.0.0.1:3000` during UI smoke; API compile/tests passed.
